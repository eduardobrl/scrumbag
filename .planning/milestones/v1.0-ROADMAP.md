# Milestone v1.0: Squad Planner MVP

**Status:** ✅ SHIPPED 2026-06-04
**Phases:** 01-06
**Total Plans:** 16

## Overview

Squad Planner v1.0 — the initial MVP. A local-first web app for planning and tracking a squad release from scope definition through sprint execution. Runs on localhost, uses SQLite, and includes MCP tools plus an AI assistant for grounded planning intelligence.

## Phases

### Phase 01: Local Foundation And Squad Setup

**Goal**: A user can run the local app, navigate the main shell, persist data in SQLite, configure core settings, and maintain squad/calendar data that future capacity calculations depend on.
**Depends on**: None
**Plans**: 3 plans

Plans:

- [x] 01-01: Walking skeleton — scaffold Next.js/SQLite app, app shell, and persisted squad member loop
- [x] 01-02: Persisted Settings screen and API for capacity defaults and MCP configuration
- [x] 01-03: Squad and calendar management for members, absences, holidays, and summary metrics

**Details:**
Delivered a localhost-only Next.js app with side navigation, global header, SQLite persistence (`./data/squad-planner.db`), Settings screen with capacity defaults, and full Squad screen with member create/edit/activation, absence tracking, holiday registration, and daily gross capacity metrics. Cross-cutting constraints: local-only, no auth, desktop-first operational UI.

### Phase 02: Release And Sprint Planning Core

**Goal**: A user can create a release, have sprints generated automatically, inspect the release and sprint list, and adjust sprint dates/goals safely.
**Depends on**: Phase 01
**Plans**: 3 plans

Plans:

- [x] 02-01: Release creation slice with generated persisted sprints and schema sync
- [x] 02-02: Release list, detail, edit, sprint reconciliation, and active release header context
- [x] 02-03: Sprint list, detail, goal/date editing, overlap blocking, gap warnings, and planning-summary placeholders

**Details:**
Delivered release CRUD with single-active-release guard, business-day sprint generation (sequential, non-overlapping, final sprint absorbs remaining days), sprint list/detail with period/status/capacity placeholders, and date editing with overlap prevention and gap warnings. Cross-cutting: status enum (IN_PROGRESS/PLANNED/CLOSED), generated sprints calculated from business days.

### Phase 03: Feature, Story, And Backlog Planning

**Goal**: A user can break release scope into features and stories, keep estimates aggregated automatically, and move unplanned stories from backlog into selected sprints with impact preview.
**Depends on**: Phase 02
**Plans**: 3 plans

Plans:

- [x] 03-01: Feature and Story data foundation, feature CRUD, cancellation, and aggregate metrics
- [x] 03-02: Story CRUD from feature detail with estimates, status, cancellation, and aggregate recalculation
- [x] 03-03: Backlog filters, story-to-sprint planning preview, sprint assignment, return-to-backlog, and planned-effort updates

**Details:**
Delivered feature/story CRUD with non-destructive cancellation, automatic feature aggregates (status, progress, story points, estimated days), backlog list with multi-facet filters, sprint planning preview showing capacity impact, move-to-backlog behavior, and sprint planned-effort summaries. Cross-cutting: cancellation statuses over destructive deletion, canceled stories excluded from totals.

### Phase 04: Sprint Board, Capacity Engine, And Leakage

**Goal**: A user can execute a sprint on a drag-and-drop board while the system calculates real capacity, highlights over-capacity plans, closes/reopens sprints, and records leaked stories.
**Depends on**: Phase 03
**Plans**: 3 plans

Plans:

- [x] 04-01: Sprint board foundation: fixed columns, HTML5 drag-and-drop, and backlog story assignment with capacity preview
- [x] 04-02: Capacity engine: gross/net capacity from members, absences, holidays, meeting/support percentages, 8-hour day normalization, and over-capacity warnings
- [x] 04-03: Sprint closure and leakage: close/reopen rules, unfinished story migration, auto-create next sprint, leakage history recording, and board read-only mode

**Details:**
Delivered 3-column sprint board (Backlog da Sprint/Em Execucao/Finalizado) with drag-and-drop status updates, full capacity calculation pipeline (active members × business days × hours − absences − holidays → net capacity → 8-hour normalized days), occupancy alerts, sprint close/reopen with unfinished story migration, automatic next-sprint creation, append-only leakage history, and closed-sprint read-only board mode. Cross-cutting: capacity calculated on demand, leakage append-only, story status transitions consistent between board and closure.

### Phase 05: Release Intelligence, Reports, MCP, And AI

**Goal**: A user can understand release health through dashboard/timeline/reporting, export planning data, and ask a local AI assistant for MCP-grounded analysis and controlled suggestions.
**Depends on**: Phase 04
**Plans**: 3 plans

Plans:

- [x] 05-01: Dashboard, timeline, and progress with real metrics, alerts, sprint table, and release timeline
- [x] 05-02: Reports and exports — 7 report types from live data with CSV and Excel export
- [x] 05-03: MCP server and AI assistant — local MCP tool surface, AI chat, quick prompts, and dangerous-operation confirmation

**Details:**
Delivered dashboard cards (progress, capacity, risk, feature/story counts, leaks), consolidated alert engine (7 alert types), release timeline showing feature spans across sprints, 7 report types (planning, capacity, sprint-story, feature-progress, leakage, planned-vs-capacity, timeline) with CSV/Excel export, localhost MCP HTTP server with read/suggest/write/safe tool categories, AI assistant chat with quick prompts, and confirmation gates for sensitive operations. Cross-cutting: dashboard/report/MCP query logic shared, dangerous MCP/assistant actions require explicit confirmation.

### Phase 06: UX Polish And Localization

**Goal**: Polish the UI for daily use: sprint durations display calendar days + business day count, global release view switcher, larger action buttons with tooltips, and full pt-BR localization.
**Depends on**: Phase 05
**Plans**: 1 plan

Plans:

- [x] 06-01: Calendar-day sprint display, global release switcher, larger action buttons, and full pt-BR i18n

**Details:**
Delivered pt-BR default locale with English fallback using next-intl, global release view switcher in the header (driven by releaseId URL param, preserves page context), sprint calendar ranges with business-day counts, shared 40px icon action buttons with tooltips and aria labels, and localized app shell plus all major operational screens. Cross-cutting: release switching is view-only (no status change), calendar-day display is cosmetic, pt-BR is default.

## Milestone Summary

**Decimal Phases:** None

**Key Decisions:**

| Decision | Rationale | Outcome |
|----------|-----------|---------|
| Use Prisma 7 with better-sqlite3 adapter | Node 24 host compatibility | Accepted — Phase 1 |
| Prepare SQLite with local sync script | Schema apply failed on this Windows/Node 24 host | Accepted — Phase 1 |
| Keep sprint assignment behind backlog planning preview | Direct story edits should not bypass capacity-impact context | Accepted — Phase 3 |
| Calculate capacity on demand from local planning data | Consistency without a separate summary table | Accepted — Phase 4 |
| Keep leakage history append-only | Preserve historical truth for reports and dashboards | Accepted — Phase 4 |
| Share dashboard/report/MCP query logic | Consistency across UI, exports, and AI/tool surfaces | Accepted — Phase 5 |
| Gate dangerous MCP and assistant actions with confirmation | Sensitive planning changes remain explicit user actions | Accepted — Phase 5 |
| Treat release switching as view context | Inspect any release without changing active-release status | Accepted — Phase 6 |
| Keep business-day sprint display cosmetic | Capacity math unchanged | Accepted — Phase 6 |

**Issues Resolved:**
- Prisma 7 + better-sqlite3 compatibility on Windows/Node 24 — solved with local db:sync script
- Backlog-to-sprint planning preview avoids bypassing capacity impact
- Leakage history preserves truth across sprint close/reopen cycles
- Dashboard, reports, and MCP share query logic for consistency
- Release view switching keeps the single-active-release rule intact

**Issues Deferred:**
- Impediment model (IMP-01 to IMP-03) — explored but not implemented in v1.0
- WIP limits on board columns — deferred to v2
- Story assignee ownership — deferred to v2
- Automatic velocity calculation — deferred to v2
- Multi-user auth — deferred to v2

**Technical Debt Incurred:**
- Prisma schema validate via local sync script instead of native migrate — may need revisiting when Node/Prisma versions mature
- Capacity is calculated on demand rather than cached — fine for v1 scale, may need optimization for large datasets
