Output: ReleasePlanningService, allocation repository methods, board summary route, and allocation update route.
</objective>

## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| Browser to release planning API | Feature allocation and ordering mutate release planning state |
| Service to CapacityService | Capacity values influence warning display and future planning decisions |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-04-06 | Tampering | Feature span uses a sprint from another release | mitigate | Repository validates both sprint ids belong to the release |
| T-04-07 | Tampering | Inverted feature span corrupts timeline | mitigate | Validate sprint ordering before update |
| T-04-08 | Denial of Service | Capacity calculation failure breaks board summary | mitigate | Service degrades to warning-friendly summary rather than throwing in UI path |
| T-04-09 | Information Integrity | Story points treated as days | mitigate | Service uses estimate_days for capacity math and only displays points as metadata |
</threat_model>
