## Spacing Scale

Declared values (must be multiples of 4):

| Token | Value | Usage |
|-------|-------|-------|
| xs | 4px | Inline metadata gaps, compact badge padding, table cell micro-gaps |
| sm | 8px | Button inner gaps, card child spacing, tab underline spacing |
| md | 16px | Default form, card, column, and table padding |
| lg | 24px | Page sections, release header, board toolbar |
| xl | 32px | Two-panel release planning layout gap, major vertical rhythm |
| 2xl | 48px | Empty states, board loading/skeleton zones |
| 3xl | 64px | Page-level separation on wide viewports only |

Exceptions: none. Interactive controls that need larger hit areas must be composed from the declared scale, such as `min-height: 48px` using `2xl`.

## Typography

| Role | Size | Weight | Line Height |
|------|------|--------|-------------|
| Body | 14px | 400 | 1.5 |
| Label | 12px | 600 | 1.2 |
| Heading | 20px | 600 | 1.2 |
| Display | 28px | 600 | 1.2 |

Only two weights are allowed: regular `400` and semibold `600`. Do not introduce bold `700` during this phase; current `font-bold` page title should be replaced or isolated to legacy code when the release screens are implemented.

## Color

| Role | Value | Usage |
|------|-------|-------|
| Dominant (60%) | `gray-50` / `#f9fafb` | Page background and low-emphasis canvas behind planning tools |
| Secondary (30%) | `white` / `#ffffff` with `gray-200` borders | Cards, release panels, tables, sprint columns, forms, tab content |
| Accent (10%) | `blue-600` / `#2563eb` | Primary CTA, active top-level tab, selected release/sprint row, keyboard focus ring, predicted completion marker |
| Destructive | `red-600` / `#dc2626` | Delete release, remove feature from release, delete sprint, delete backlog item |

Accent reserved for: primary release action, active navigation state, selected planning row, focus ring, predicted completion marker, and selected timeline span border. Do not use accent for every interactive element; secondary buttons remain gray, success/status chips remain semantic green/blue/gray, and warnings remain amber.

Warnings: use `amber-50` background, `amber-200` border, and `amber-800` text for over-capacity and oversized-feature suggestions.

## Interaction and Visual Hierarchy

Primary focal point: the release planning board/timeline. The first viewport must show the selected release name, release status, sprint columns, at least one feature lane or empty lane, and capacity indicators without making the user scroll past generic backlog CRUD.

Release detail layout:

| Area | Contract |
|------|----------|
| Header | Display release name at 28px, status chip, date range, and release capacity summary. Primary CTA is `Criar sprint da release`. |
| Tabs | Use text tabs: `Features`, `Sprints`, `Capacity`, `Forecast`. Default tab is `Features`. Active tab uses the accent border and text color. |
| Feature backlog panel | Left panel on desktop, top section on mobile. Shows available features with title, estimate days, story/bug count, and optional epic label. |
| Timeline | Main area uses sprint columns. Feature rows span one or more sprint columns using CSS grid based on sprint ids, not pixel positions. |
| Feature span controls | Drag/drop is allowed for assignment and ordering, but every span must also have keyboard-accessible controls to move start/end sprint. |
| Capacity feedback | Sprint column headers show available days, planned days, and warning state. Over-capacity is visible but non-blocking. |
| Split suggestion | Oversized features show an amber suggestion affordance: `Sugerir divisao da feature`, opening a lightweight guided prompt or prefilled feature form. |
| Sprint drill-down | Clicking a sprint opens a dedicated sprint screen, not inline content below the list. Sprint screen tabs are `Board`, `Planning`, `Capacity`, `Closure`. |

Mobile behavior: stack header, tabs, feature backlog, and timeline vertically. Timeline may horizontally scroll within its own region, but headers and capacity warnings must remain readable at 14px.

## Copywriting Contract

| Element | Copy |
|---------|------|
| Primary CTA | Criar sprint da release |
| Secondary CTA | Adicionar feature a release |
| Feature creation CTA | Criar feature |
| Story/bug creation CTA | Adicionar historia ou bug |
| Empty state heading | Nenhuma feature nesta release |
| Empty state body | Crie uma feature ou arraste uma feature existente para planejar escopo, capacidade e previsao por sprint. |
| Release board empty sprint | Nenhum sprint nesta release. Crie o primeiro sprint para montar a timeline. |
| Error state | Nao foi possivel atualizar o planejamento da release. Revise a feature, o sprint de destino e tente novamente. |
| Capacity warning | Escopo acima da capacidade deste sprint. Ajuste o intervalo da feature ou divida o trabalho. |
| Oversized feature suggestion | Esta feature parece grande demais para a release. Considere dividir em features menores. |
| Destructive confirmation | Excluir release: Excluir esta release remove o agrupamento de planejamento. Sprints e features afetados devem ser revisados antes de continuar. |
| Destructive confirmation | Remover feature da release: Remover esta feature da release tambem remove sua alocacao na timeline. |
| Destructive confirmation | Excluir sprint: Excluir este sprint remove sua coluna da release e exige revisar as features alocadas. |

Do not use generic labels such as `Submit`, `OK`, `Save`, or `Cancel` in new Phase 4 UI. Existing legacy labels can remain until touched, but new release screens must use action-specific copy.
